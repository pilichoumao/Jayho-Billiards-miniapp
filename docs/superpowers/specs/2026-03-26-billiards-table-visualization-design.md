# 台球桌可视化与路线绘制设计

日期：2026-03-26  
状态：设计稿

## 1. 目标

在现有 `Jayho-Billiards-miniapp` MVP 基础上增强首页表现，使用户可以：

1. 看到完整的台球桌可视化。
2. 查看求解结果对应的路线图。
3. 拖动球位后保留编辑态，不自动计算。
4. 点击 `Calculate` 时才基于当前球位重新求解。
5. 在候选结果之间切换高亮，但不触发重新求解。

本次增强只做“展示与交互层可用”，不改变核心求解器的几何逻辑，不引入实时预览求解。

## 2. 已确认约束

1. 用户修改球位置后，不实时计算。
2. 用户再次点击 `Calculate` 时，才重新生成结果。
3. 默认只展示当前选中的一条候选线路。
4. 候选切换只影响高亮，不影响编辑态球位。
5. 现有核心层已经输出 `segments`、`cushions`、`metrics`，前端应直接复用这些数据。

## 3. 设计原则

1. 编辑态与结果态分离。
2. 以现有 `SolveRequest` / `SolveResponse` 为唯一求解接口。
3. 桌面坐标继续使用归一化坐标，UI 只负责映射到像素。
4. 默认信息密度可控，避免同时展示太多候选导致画面拥挤。

## 4. 页面结构

首页拆成三层：

1. 顶部说明区
   - 保留品牌和模式说明。
2. 中部台面区
   - 绘制台球桌、球体、当前候选路线。
3. 底部结果区
   - 显示候选摘要，支持切换高亮。

页面仍然保留现有模式切换与 `Calculate` 按钮，但按钮只在用户明确点击时触发求解。

## 5. 状态模型

页面状态分为两类：

```ts
type EditState = {
  balls: Ball[];
  selectedBallId?: string;
};

type SolveState = {
  requestMode: SolveMode;
  response?: SolveResponse;
  selectedCandidateId?: string;
  errorText?: string;
};
```

编辑态负责：

1. 维护当前拖拽后的球位。
2. 记录当前被拖动的球。
3. 作为下一次求解输入。

结果态负责：

1. 保留最近一次求解结果。
2. 记录当前选中的候选线路。
3. 存放错误信息或无解提示。

这两个状态互相独立。拖拽只改编辑态，结果只在点击计算时刷新。

## 6. 画布渲染

### 6.1 台面

台面使用固定比例容器渲染，保持和核心层相同的归一化坐标体系。UI 侧只做像素映射，不在前端再定义一套物理坐标系。像素映射采用统一缩放因子，保证球和袋口在屏幕上仍保持圆形，不因宽高比被拉伸。`Table.width`/`Table.height` 只用于确定台面长宽比和容器尺寸，不参与球半径或袋口半径的换算。

台面元素包括：

1. 台呢背景。
2. 库边边框。
3. 袋口或边角提示。
4. 可选的轻量辅助网格或标尺。

### 6.2 球体

球体渲染分为三种角色：

1. 母球
2. 目标球
3. 障碍球

球体支持拖动，但仅限编辑态。拖动后立即更新画面中的球位，不触发求解。

### 6.3 路线

当前选中的候选线路根据 `segments` 绘制折线：

1. `start` 段作为起始线段。
2. `cushion` 段表示碰库过程。
3. `contact` 段表示接触目标球或命中目标。
4. `end` 段表示终止位置。

关键事件点需要额外标记：

1. 碰库点。
2. 接触点。
3. 终点或超时终止点。

默认只绘制一条候选线路，保持画面清晰。其他候选仅在结果面板中展示摘要。

## 7. 交互规则

### 7.1 拖动

1. 用户按住球并拖动时，只更新编辑态球位。
2. 松手后不自动计算。
3. 若拖动导致当前结果过期，旧结果保留，直到用户下一次点击 `Calculate`。

### 7.2 计算

1. 点击 `Calculate` 时读取当前编辑态球位。
2. 生成新的 `SolveRequest`。
3. 调用 `solveShot`。
4. 成功后替换结果态。
5. 失败时保留编辑态，显示错误。

### 7.3 候选切换

1. 默认选中排序后的第一条可行候选。
2. 用户点击其他候选时，只切换高亮。
3. 切换候选不触发求解，不修改球位。

## 8. 结果面板

结果区保留现有摘要信息，但内容调整为更适合桌面可视化：

1. 候选编号或名称。
2. 库数。
3. 总路程。
4. 最小净距。
5. 拒绝原因或可行状态。

如果当前结果没有可行候选，则结果区显示无解提示，桌面仍显示编辑态球位。

## 9. 数据流

1. 用户编辑球位。
2. 编辑态更新，桌面立即重绘。
3. 用户点击 `Calculate`。
4. 当前编辑态转换为 `SolveRequest`。
5. 求解器返回 `SolveResponse`。
6. 结果态更新，桌面绘制选中线路，结果面板显示候选摘要。
7. 用户切换候选，只更新高亮，不重新求解。

## 10. 测试策略

1. 页面状态测试
   - 拖动只改编辑态，不触发求解。
   - 点击 `Calculate` 才会更新结果态。
2. 渲染测试
   - 台面坐标从归一化坐标正确映射到画布区域。
   - 路线与球位能稳定重绘。
3. 交互测试
   - 候选切换只影响高亮。
   - 结果过期时仍保留编辑态。
4. 回归测试
   - 模式1和模式2都能正常显示桌面和线路。

## 11. 非目标

本次增强不包含：

1. 拖动时实时求解。
2. 动画化回放。
3. 多候选同时叠加展示。
4. 更复杂的物理拟真升级。
5. 后端接口改造。

## 12. 交付标准

满足以下条件即可认为本次增强完成：

1. 首页能显示台球桌与球位。
2. 求解后能显示当前选中候选的路线。
3. 用户拖动球后不会自动重算。
4. 用户点击 `Calculate` 后会基于新球位重新生成结果。
5. 候选切换会同步高亮，但不触发重算。
