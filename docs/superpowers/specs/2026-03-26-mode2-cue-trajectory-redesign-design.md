# Mode 2 Cue Trajectory Redesign Design

日期：2026-03-26  
状态：设计稿

## 1. 目标

将 `mode2_cue_direction` 从“带目标球的方向求解”调整为“纯母球轨迹预览”：

1. 页面只展示母球，不再展示目标球或障碍球。
2. 用户可以拖动母球位置。
3. 用户可以通过箭头手柄设置母球初始方向。
4. 点击 `Calculate` 后才重新计算。
5. 路线必须以连续线段绘制，而不是只标点。
6. 台面必须显示 6 个袋口。
7. 轨迹如果进入袋口区域，solver 要返回截断到袋口边界精确交点的轨迹，前端只负责按返回数据绘制并标记为无效。
8. `mode2` 只展示最多 4 库内的母球行进路线。

本次设计不改变 `mode1_contact_paths` 的现有目标和交互，只收紧 `mode2` 的语义与展示。

## 2. 已确认约束

1. `mode2` 只展示母球，台面上也只有母球。
2. 母球初始方向由箭头手柄设置。
3. 拖动母球或箭头时，不自动计算。
4. 只有再次点击 `Calculate` 才会产生新结果。
5. 路线必须画成连续线段。
6. 袋口必须可视化。
7. 进入袋口的线路不能作为可用结果。
8. 如果线路进入袋口，solver 必须返回截断到袋口边界交点的最后一段，前端只负责按返回数据绘制并标记无效。
9. `mode2` 只展示 4 库以内的路线。

## 3. 设计原则

1. 编辑态与结果态分离。
2. `mode2` 的输入只包含母球位置和初始方向。
3. UI 负责展示袋口、拖拽和路线高亮。
4. solver 负责生成轨迹与判定无效轨迹。
5. 展示层不应再自行拼装“假结果”来掩盖无效轨迹。
6. 所有 `mode2` 几何都使用与现有首页一致的归一化台面坐标 `[0,1]`，UI 只做像素映射。
7. `Ball.radius` 和 `SolveRequest.table.pocketR` 都是归一化单位的无量纲半径，取值与 `[0,1]` 台面坐标同域；`table.width`/`table.height` 只用于台面比例和视觉缩放，不参与半径换算。
8. 页面渲染使用统一缩放因子，x/y 方向共用同一个像素比例，保证球和袋口保持圆形。

## 4. 页面结构

首页中部台面区拆成四层：

1. 台面底图
   - 画台呢、库边和 6 个袋口。
2. 结果轨迹层
   - 画连续折线、反弹点、截断点和无效标记。
3. 编辑层
   - 画可拖动的母球与方向箭头。
4. 控件层
   - 保留模式切换、`Calculate` 按钮和结果摘要。

在 `mode2` 下，页面只渲染母球和箭头，不渲染目标球或障碍球。页面仍然沿用首页的 `balls[]` 作为编辑态来源，`mode2` 里该数组只保留一颗 `cue` ball；`cueDirection` 作为同一页里的附加输入保存，但不改变首页 `EditState` 的基本形状。

## 5. 状态模型

页面状态分为两类：

```ts
type EditState = {
  balls: Ball[];
  selectedBallId?: string;
};

type Mode2InteractionState = {
  cueDirection: Vec2;
  draggingHandle?: "ball" | "direction";
};

type PageState = EditState & Mode2InteractionState;

type SolveState = {
  requestMode: SolveMode;
  response?: SolveResponse;
  selectedCandidateId?: string;
  errorText?: string;
};
```

编辑态负责：

1. 维护当前母球位置。
2. 维护当前初始方向。
3. 记录正在拖动的是母球还是方向箭头。
4. `cueDirection` 在页面状态里始终保存为归一化单位向量，不等到提交时才归一化。

结果态负责：

1. 保留最近一次求解结果。
2. 记录当前选中的轨迹候选。
3. 存放错误信息、无效轨迹提示或无可用路线提示。

拖拽只改编辑态，点击 `Calculate` 才刷新结果态。`cueDirection` 作为 mode2 的附加页面字段单独保存，但它属于页面状态的一部分，而不是 `balls[]` 中的球对象；页面状态中的 `cueDirection` 始终为归一化单位向量。
在 mode2 中，页面状态 = `EditState` + `Mode2InteractionState`，而不是新的 cue-only edit-state。

## 6. 轨迹与袋口规则

### 6.1 连续线路

`mode2` 的主视觉必须是连续折线：

1. 每个 `segment` 直接连成一条完整轨迹。
2. `cushion` 点保留为辅助标记。
3. `start` 和终点保留辅助点。
4. 线路不能只画散点，不能只在库边标记命中点。

### 6.2 袋口显示

台面上必须显示 6 个袋口，位置由归一化台面的四角和中点决定，袋口半径直接复用 `SolveRequest.table.pocketR`。归一化坐标系保持现有首页约定：左上角为 `(0, 0)`，x 向右，y 向下。

1. 左上角 `(0, 0)`
2. 中上 `(0.5, 0)`
3. 右上角 `(1, 0)`
4. 左下角 `(0, 1)`
5. 中下 `(0.5, 1)`
6. 右下角 `(1, 1)`

袋口既是视觉元素，也是轨迹判定的边界。袋口区域定义为“以袋口中心为圆心、半径等于 `table.pocketR` 的圆”。
球心轨迹与袋口圆的判定都在归一化坐标中进行，不再引入额外的物理单位换算。

### 6.3 进袋处理

如果轨迹进入袋口区域：

1. solver 在每一段推进时，分别计算 6 个袋口圆与当前轨迹线段的交点，并取所有袋口中最小的正参数 `tPocket`；同时计算母球球心在可放置区域边界上的最近交点参数 `tCushion`。若 `tPocket` 存在且 `tPocket <= tCushion + EPSILON`，则以该最早袋口交点为准并截断，不再继续反弹。
2. 袋口处显示截断或无效标记。
3. 该候选标记为无效，`blocked=true` 且 `rejectReason="pocketed"`。
4. 最后一段仍使用现有 `PathSegment.event`，并保留为 `end`，由 `blocked/rejectReason` 表达“这是进袋截断而不是正常终止”。
5. 结果区展示无效原因，不把它当作可用路线。

这比“只要最终进袋就完全不画”更利于用户理解轨迹是如何失败的。前端不自行计算交点，只使用 solver 返回的截断线段。

### 6.4 4 库上限

`mode2` 只展示最多 4 库内的行进路线：

1. solver 在第 4 次库边反弹后停止继续传播，这属于正常终止条件。
2. 如果轨迹在第 4 次库边反弹后还会继续发生第 5 次反弹，solver 仍只返回前 4 库的折线路径，不额外标记为无效。
3. 结果区要能说明这是“最多 4 库”的正常上限。
4. UI 不需要同时叠加多条候选，只显示当前选中的一条。

## 7. 交互规则

### 7.1 母球拖动

1. 用户拖动母球时，只更新母球位置。
2. 松手后不自动计算。
3. 母球中心必须保持在归一化台面的可放置区域内：`x` 和 `y` 都要落在 `[radius, 1 - radius]`，并且母球中心不能落入任一袋口圆内。这里的“库边交点”与可放置区域边界一致，指母球中心轨迹在 `x = radius`、`x = 1 - radius`、`y = radius`、`y = 1 - radius` 四条边上的交点。

### 7.2 方向箭头

1. 箭头手柄与母球绑定显示。
2. 用户拖动箭头时，只更新初始方向。
3. 方向箭头长度可以固定，角度由拖拽方向决定。
4. 箭头指向就是 `cueDirection`。
5. 当方向箭头在归一化台面中的拖拽长度小于 `0.02` 时，页面保留上一次有效方向，不向 solver 提交零向量。

### 7.3 计算

1. 点击 `Calculate` 时读取当前母球位置和初始方向。
2. 生成新的 `SolveRequest`。
3. 调用 `solveShot`。
4. 成功后替换结果态。
5. 失败时保留编辑态，显示错误。

## 8. `mode2` 请求语义

`mode2_cue_direction` 仍然使用现有的 `SolveRequest` 结构，不引入新请求类型。

用户可编辑的输入只有母球位置和初始方向；`table` 和 `constraints` 仍然是求解请求的必填字段，但不新增独立的 UI 输入控件。

`mode2_cue_direction` 的请求只保留母球和方向：

1. `balls` 只包含 `cue`。
2. 不再要求 `target`。
3. 不再依赖障碍球。
4. `table` 必填，且必须包含 `width`、`height` 和 `pocketR`，用于台面缩放和袋口判断；`constraints` 也必须保留，`avoidObstacle=true`，`timeoutMs` 继续生效，`cushionMin`/`cushionMax` 在 mode2 中可以省略或忽略。
5. `input.cueDirection` 必填，且作为归一化后的单位向量存储；方向箭头的视觉长度可以固定，但写入请求前必须先归一化。
6. 轨迹的所有几何判断继续使用归一化坐标，和首页现有台面坐标约定保持一致。

这个模式的目标是展示母球行进路线，因此请求和结果都不应再携带与目标球有关的视觉假设。

## 9. 结果面板

结果区保留少量、明确的信息：

1. 母球初始方向。
2. 库数。
3. 总路程。
4. 是否有效。
5. 无效原因。

如果没有可用路线：

1. 桌面仍显示编辑态母球和方向箭头。
2. 结果区显示无可用路线提示。
3. 不显示伪造的候选路线。

## 10. 数据流

1. 用户拖动母球或方向箭头。
2. 编辑态更新，桌面立即重绘。
3. 用户点击 `Calculate`。
4. 当前编辑态转换为 `SolveRequest`。
5. solver 返回 `SolveResponse`。
6. 页面绘制当前候选的连续轨迹。
7. 如果轨迹进入袋口，页面只渲染 solver 返回的截断结果并标记无效。
8. 如果轨迹达到第 4 次库边反弹，页面保留截至该点的折线路径作为正常上限结果。

## 11. 测试策略

1. 页面状态测试
   - 拖动母球和方向箭头不会触发求解。
   - 点击 `Calculate` 才会刷新结果态。
2. 渲染测试
   - `mode2` 只显示母球与方向箭头。
   - 台面上有 6 个袋口。
   - 路线以折线绘制，不是散点。
3. 交互测试
   - 方向箭头拖动会更新 `cueDirection`。
   - 进入袋口的路线会被 solver 截断到袋口边界交点，并标记无效。
   - 超出 4 库的路线会在第 4 次反弹后停止并作为正常上限结果返回。
4. 回归测试
   - `mode1` 现有行为不受影响。
   - `mode2` 不再要求目标球。

## 12. 非目标

本次增强不包含：

1. 拖动时实时求解。
2. 动画化回放。
3. 多条候选同时叠加展示。
4. 旋转、摩擦和能量损失建模。
5. 后端接口大改造。

## 13. 交付标准

满足以下条件即可认为本次增强完成：

1. `mode2` 页面只显示母球和方向箭头。
2. 台面能显示 6 个袋口。
3. 轨迹以连续线段绘制。
4. 轨迹进入袋口时会被截断并标记无效。
5. `mode2` 只展示最多 4 库内的路线；超过 4 库时只返回前 4 库的正常上限结果。
6. 拖动母球或箭头后不会自动重算。
7. 点击 `Calculate` 后会基于新位置和新方向重新生成结果。
